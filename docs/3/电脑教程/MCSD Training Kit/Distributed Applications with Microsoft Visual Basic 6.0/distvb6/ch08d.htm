Summary

A stored procedure is a precompiled database object that can include validation, modification, insertion, and deletion of data. Stored procedures are beneficial because they can encapsulate business rules and logic that can be reused by many applications. Stored procedures can also hide complex SQL statements from applications, and also from developers, who typically do not need to know the structure of the database.

Stored procedures work well in a two-tier model because there is minimal network traffic due to the low number of roundtrips between a client and server. Stored procedures can also be used in an n-tier model. In an n-tier model, a client application can call a method in a COM DLL that in turn calls a stored procedure on a database server. You can use the Execute method of an ADO Command object to run a stored procedure. If the stored procedure returns records, an ADO Recordset object is returned from the Execute method.

An application communicates with a stored procedure through parameters. Your application can pass data to a stored procedure by using ADO Parameter objects, which represent stored procedure parameters. In addition, output parameters can be used to allow a stored procedure to return a data value or Recordset back to an application. You can also implement transactions in stored procedures to group together SQL commands as a single unit of work using the Begin Transaction and Commit statement. Transactions in a stored procedure can also be nested. You can use the T-SQL debugger to interactively debug stored procedures written in SQL Server from within the Visual Basic development environment.
