1、You are the branch administrator. When you ping the computer that shares the Internet access for your branch office, you get a  positive response.  However, you are unable to access the Internet from your branch office computer. What should you do to assure that the IP address you think is assigned to the Internet share computer is the actual one?

1.  Check the current mappings of your computer with the Route command.

2.  Check the TCP/IP connections using Nbtstat on your computer.

3.  Run REXEC from your computer to see if the target computer is able to access the Internet.

4.  Run ARP command using the hardware address of the target computer's adapter. <Correct>

Explanation :

If you get the hardware address of the target computer's adapter and run ARP from your computer, you'll see if the IP address that was resolved really does map to the physical address of the target computer. The problem may be that another computer actually has the IP address that you think belongs to the Internet share computer.

Route allows you to manage your local routing table.

Ntstat will give you information about NetBIOS over TCP/IP.

REXEC allows you to run a process on a remote computer.

 

2、A user sent a Postscript file to a PCL printer and now reports that each page printed has one character on it and won't stop printing.  What is your first line of action to quickly correct the problem?

1.  Ask the user to Purge the job from the Printer's queue.

2.  Pause the printer, in the Printers folder, from the context menu. <Correct>

3.  Ask the Print Operator to press the offline button on the printer.

4.  Select Purge Print Documents from the Printer pull-down menu.

Explanation :

Pause is important here so you can manage the problem without ongoing printing.  To fix this problem you must delete the print job if you have permission, or ask the user to delete it. You can then fill the paper tray with these used sheets of paper and let the printer print what's remaining in memory.  A Postscript file printed on a PCL printer will often print one character per page. Printers that cannot cancel the job, including what is in memory, will use lots of paper.

2.  You tell him to boot with the Emergency Repair Disk (ERD) that is stored at each user's desk.  You think he probably has some corrupt system files that ERD can restore very easily.

3.  You walk him through a Last Known Good Configuration because it is the easiest troubleshooting solution, regardless of the circumstances before the system failure.

13、You notice that over a two week period that a processor queue is developing.  You have a single processor running at about 90 percent..  About one-third of the time the queue has one or two threads waiting.  You must decide if this is due to random threads requesting processor time simultaneously or if some of the processes have threads that run for irregular amounts of time.  What should you do next to determine the properties of these threads?

1.  Run CpuStress to identify the threads by process and instances.

2.  Run the Task List Viewer (Tlist.exe) utility to identify the threads by process and instances.

3.  Use the System Calls/sec to reveal what part of the processor usage is given to privileged mode.

4.  Log the counters of the Processor, Process, and Thread objects to associate thread activity with processor usage. <Correct>

Explanation :

Your goal in analyzing Processor, Process, and Thread objects is to determine if your bottleneck is caused by multiple processes or multithreaded processes.  You can improve performance for multiple processes by adding another processor.  If you have a multithreaded process causing a bottleneck, then you have to further determine if multiple threads are actually running simultaneously.  If they are, you can perhaps get a performance improvement with another processor.  If the multithreads are inactive most of the time, you will not get improvement from another processor. 

The Tlist utility provides thread identifier information, not thread instances or association with a process.

CpuStress is a utility that runs a test to analyze processor activity.

You can analyze privileged mode versus user mode after you associate thread activity with processor usage.  This analysis is something developers use when optimizing an application's performance.

15、You know that the processor is overwhelmed and you already know that a single process is causing the problem.  Where can you find out whether to purchase another processor or a faster processor?

1.  In System Monitor, histogram view, Processor(_Total)\% Processor Time

2.  In perfmon.msc, %Processor Time values for the System process

3.  In the Pviewer utility report of Thread(_Total)\Context Switches/sec

4.  In Task Manager, Processes tab, Columns view of Thread Count <Correct>

Explanation :

Task Manager allows you to view the Thread Count of a process and determine whether it has single or multiple threads.  For a multiple thread process, you will optimize with another processor.  For a single thread process, you will optimize by purchasing a faster processor.  The Task Manager Thread Count is an interim step. You will then collect data in System Monitor on the properties of the thread(s).

The System Monitor histogram view will show you the processes using the CPU time and the percentage being consumed by each process.  Use this step to determine if a single or multiple processes are monopolizing the processor.

Perfmon.msc is the command line version of System Monitor.  In Windows 2000, you no longer use the system object.  You'll need to measure total processor time with the _Total instance of the Processor object.  The syntax looks like this: Processor(_Total)%Processor Time.  Notice that NT's % Total Processor Time is now %Processor Time for Windows 2000.

The Process Viewer (pviewer.exe ) utility, which you can install from on the Windows 2000 operating system CD,  reports context switch data.  A context switch occurs when the kernel switches the processor from one thread to another.  The pviewer result shows processor time as 100% because idle time is included with user and privileged modes to total 100%. If you use System Monitor, however, you will see only the processor time actually used for user and privileged modes and no idle time.
